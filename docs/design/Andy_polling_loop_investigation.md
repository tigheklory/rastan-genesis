## 0. Executive verdict

**The framing was wrong.** The loop at `0x0003A192 ↔ 0x0003A19C` is NOT a polling wait. It is a **watchdog-style countdown delay followed by an unconditional soft-reset to bootstrap**. The BM-002 byte reading (`66 F4` at `0x0003A19C` = `BNE.S -12`) is independently CONFIRMED. The BM-002 characterization of the surrounding region as "delay loop / soft-reset preamble" was correct in essence; what Andy now adds is the routine's full structural shape and its role in the project's overall failure mode.

**Architectural read: this routine IS the OPEN-004 bootstrap-re-entry mechanism.** A short prologue at `0x0003A180` checks an in-RAM "watchdog timer" at `%a5@(44)` (= `0x00FF002C`). If positive, the routine decrements and returns immediately (the watchdog is "alive"). If zero, the routine runs ~655,360 iterations of a delay loop (~3.6s on Genesis 68000), reloads SP and PC from the reset vectors at `0x00000000`/`0x00000004`, and `JMP`s into `_bootstrap` at `0x00000202`. **OPEN-004's "~15 re-entries in 64s" timing matches: 64s / ~4.3s per cycle ≈ ~15 cycles, where ~3.6s is the delay loop plus ~0.5-1s of bootstrap-restart code.**

**OPEN-001 and OPEN-004 collapse into the same root cause.** The watchdog has SEVEN code sites that "kick" it (write a positive value to `%a5@(44)`) and one site that explicitly clears it. The kicks are scattered through arcade-translated code at runtime PCs `0x0003A5D4`, `0x0003A63E`, `0x0003AC88`, `0x0003ACF2`, `0x0003AD22`, `0x0003AD5E`, `0x0003ADD0`. Cody's Build 0077 video shows the CPU's excursions during the 14.8s observation period visit `0x000002D4` (vector), `0x0003B0B6 / 0x0003B0E6` (arcade-translated handler region), and `0x000705F6 / 0x0007060C / 0x00070614 / 0x00070624 / 0x000719D4 / 0x000719E0 / 0x00071B60 / 0x00071CE2` (Genesis-native helper region). **None of the seven kick sites is in any of these excursion-visited regions.** The watchdog is timing out because the arcade game-loop main path that's supposed to reach the kicks isn't running — exactly the same condition Andy's nametable composition path classification flagged as the OPEN-001 proximate cause.

**Confidence: HIGH** that this is a broken watchdog (not a correctly-idle wait). Evidence: the routine's structure (unconditional soft-reset at expiry; no exit branch other than expiry), the kick sites' arcade-code locations (post-bootstrap game-loop range), the observed excursion pattern (interrupts firing but not reaching kicks), the existing nametable-composition classification (game-loop path blocked), and OPEN-004's existing 15-re-entries-per-64s data (matches the 3.6s-per-delay-cycle estimate).

**Recommended next diagnostic step (Chad routes; Andy does NOT route).** Identify which of the seven kick sites is the "primary" kick the arcade game-loop reaches under normal Rastan operation. This is the kick whose absence is causing the watchdog to expire. Most likely candidates by location: `0x0003ACF2` and `0x0003AD22` (clustered in what appears to be a main-loop init/setup region), or `0x0003ADD0` (which sets the smallest value `0x10` — suggesting "quick-cycle" expectation, i.e., the per-frame kick the VBlank handler is supposed to perform). Determining which is "primary" benefits from arcade-source disassembly cross-reference (MAME's `rastan.cpp` driver, or the local arcade Rastan disassembly under `docs/reference/mame/rastan/`). The next bookmark cycle (BM-004 or BM-005, post-OPEN-013 closure) targeting the primary kick site would confirm whether the CPU ever reaches it.

## §1 Phase 1 — Loop disassembly (Deliverable 1)

### §1.1 BM-002 byte reading verified independently

Direct hex dump of `dist/rastan-direct/rastan_direct_video_test_build_0077.bin` at file offsets `0x0003A18C..0x0003A1AF`:

```
0x0003A18C: 22 3C 00 0A 00 00    movel #655360, %d1        (set D1 = 0x000A0000)
0x0003A192: 20 38 00 00          movel $0.W, %d0           (D0 = longword at $00000000)
0x0003A196: 04 81 00 00 00 01    subil #1, %d1             (D1 -= 1)
0x0003A19C: 66 F4                bnes 0x3a192              (branch to 0x3a192 if D1 != 0)
0x0003A19E: 2E 78 00 00          moveal $0.W, %sp          (SP = longword at $00000000)
0x0003A1A2: 20 78 00 04          moveal $4.W, %a0          (A0 = longword at $00000004; = reset vector = 0x00000202)
0x0003A1A6: 4E D0                jmp (a0)                  (jump to bootstrap entry)
0x0003A1A8: 65 02                bcss 0x3a1ac              (dead code after JMP)
0x0003A1AA: 60 E0                bras 0x3a18c              (dead code; would jump to delay-init)
0x0003A1AC: 4E 75                rts                       (dead code)
```

**BM-002 byte reading is CONFIRMED.** The bytes at `0x0003A19C` are `66 F4`, decoding as `BNE.S` with signed 8-bit displacement `0xF4` (= -12); target = `(0x0003A19C + 2) + (-12) = 0x0003A192`. The branch goes back 12 bytes to `0x0003A192` if the Z flag is clear (D1 != 0).

### §1.2 The full routine spans 0x0003A180..0x0003A1AC

Cross-referencing the post-patch disassembly at `build/genesis_postpatch.disasm.txt:72720-72733`, the routine includes a **prologue** at `0x0003A180..0x0003A18A` that precedes the delay-loop init at `0x0003A18C`:

```
0x0003A180:  4A 6D 00 2C          tstw %a5@(44)                ; test workram counter at A5+0x2C
0x0003A184:  67 06                beqs 0x3a18c                 ; if zero, branch to delay-loop init
0x0003A186:  53 6D 00 2C          subqw #1, %a5@(44)           ; else decrement the counter
0x0003A18A:  4E 75                rts                          ; and return (early exit)
0x0003A18C:  22 3C 00 0A 00 00    movel #655360, %d1           ; (delay-loop init — only reached if counter was zero)
... (delay loop continues per §1.1) ...
```

**The routine's structure is a watchdog:**
- Entry at `0x0003A180`: test counter at `%a5@(44)`.
- If counter is positive: decrement it and return (the "kick the watchdog forward" caller-facing path).
- If counter is zero: fall through to the delay loop + soft-reset.

The dead code at `0x0003A1A8..0x0003A1AC` (BCS.S / BRA.S / RTS after the JMP) is unreachable in practice — the JMP at `0x0003A1A6` unconditionally diverts PC. It exists as either compiler/assembler padding or as a "should never run" defensive sentinel.

### §1.3 Function symbol identification

The post-patch disassembly does not provide function-symbol annotations for the arcade-translated region (the project's `out/symbol.txt` only carries symbols for Genesis-native helpers, not arcade-source code). The routine at `0x0003A180` is a coherent function by structure (clear entry, clear control flow, clear return points), but it has no project-assigned symbol. For brevity in this design doc, Andy refers to it as `watchdog_or_reset` (descriptive, not a project-assigned name).

## §2 Phase 2 — Wait-condition identification (Deliverable 2)

### §2.1 The "wait condition" is not a polling read

Re-reading the loop body:

```
0x0003A192:  movel $0.W, %d0        ; D0 = longword at memory address 0x00000000
0x0003A196:  subil #1, %d1
0x0003A19C:  bnes 0x3a192
```

The `MOVE.L $0.W, D0` reads four bytes from memory address `0x00000000`. Per the Genesis memory map, address `0x00000000` is in cartridge ROM (the initial SSP value, per the 68000 reset vector convention). On this project's ROM, `0x00000000..0x00000003` contains `00 FF 00 00` (= `0x00FF0000`, the initial supervisor stack pointer — verified by `verify_rastan_direct_boot_guard.py:13` `EXPECTED_SP_VECTOR = 0x00FF0000`).

So the loop reads a CONSTANT value (`0x00FF0000`) from ROM into D0 every iteration. D0 is then ignored (overwritten next iteration). **This is not a polling read.** D0's value is irrelevant; the read is bus-activity filler / cycle consumption.

**The actual exit condition** is the `BNE.S` test of the Z flag, which is set by the immediately preceding `SUBI.L #1, D1`. The loop exits when D1 reaches zero. **No external memory or hardware register can satisfy this exit early — the loop is a pure countdown.**

### §2.2 The implicit wait condition is the routine-entry counter at A5+0x2C

What the routine WAITS ON at the call-site level is the value at `%a5@(44)` (= `0x00FF002C` since `%a5` is initialized to `0x00FF0000` at boot per `boot.s:164`):

- If `%a5@(44)` is positive at routine entry: the routine returns early without entering the delay loop. The system "skips" this iteration of the watchdog.
- If `%a5@(44)` is zero at routine entry: the routine enters the delay-then-reset path. The system soft-resets.

So the meaningful "satisfying value" for the system to NOT reset is: keep `%a5@(44)` positive. This is achieved by other code writing a positive value to `0x00FF002C` regularly — the "kick the watchdog" pattern.

### §2.3 Semantic role of A5+0x2C

`%a5` = `0x00FF0000` is Genesis WRAM (project's Work RAM). The offset `+0x2C` (= 44) is into the arcade game's workram structure. Without arcade-source documentation Andy can name this offset by its disassembly behavior:

- It is decremented by `subqw #1` in multiple routines (lines 72722, 72888, 73401, 73498, 73604 of the postpatch disasm — all at addresses `0x0003A186`, `0x0003A360`, `0x0003AAB2`, `0x0003AC04`, `0x0003AD74`).
- It is set to positive values (`0xA0`, `0xD0`, `0x10`, `0x200`) by `movew #imm, %a5@(44)` at multiple sites (lines 73068, 73093, 73538, 73571, 73584, 73598, 73628 — see §3).
- It is explicitly cleared by `clrw %a5@(44)` at line 73681 (= runtime PC `0x0003AE76`).

**Role:** workram word counter at `0x00FF002C`. Its value is "ticks remaining before next forced reset." Other arcade code sets it to a positive value (a "fresh deadline"); the watchdog routine decrements it; when it reaches zero, the watchdog fires the soft-reset. This is **a software-implemented watchdog timer** — a Rastan arcade convention.

Whatever the trigger, **if execution reaches `0x0003AE76`, the system is intentionally heading for soft-reset.**

The video evidence does NOT show the CPU at or near `0x0003AE76` during the observation window. So the force-expire path is not the active mechanism for what the Build 0077 video captures. The active mechanism is "no kick reached → watchdog naturally times out."

## §5 Phase 5 — Architectural read (Deliverable 5)

### §5.1 Finding: BROKEN WATCHDOG (functionally a "broken polling wait" in essence)

The loop at `0x0003A192 ↔ 0x0003A19C` is a deliberate countdown delay inside a Rastan-arcade-original watchdog routine. The routine's design assumption is that the arcade game-loop will regularly kick the watchdog via writes to `%a5@(44)` at one of seven (+4 deeper) kick sites. When the game-loop fails to kick, the watchdog expires, runs a ~3.6s delay loop, and soft-resets to bootstrap.

**Confidence: HIGH.** Supporting evidence:
- The routine's structure (unconditional soft-reset at expiry; no exit branch other than expiry).
- The kick sites' locations (post-bootstrap arcade game-loop range, downstream of `_bootstrap` and outside any handler region).
- The observed excursion pattern (interrupts firing → handlers run → return to loop; handlers don't reach kicks).
- OPEN-004's existing "~15 re-entries in 64s" timing matches the ~3.6s delay-loop estimate (15 cycles × ~4.3s/cycle ≈ 64s).
- The nametable composition classification's independent finding that the game-loop is blocked (Andy `Andy_nametable_composition_path_classification.md`).
- The video's frame timing matches a CPU running a tight loop being yanked into interrupts.

The "correctly-idle wait" alternative (e.g., title-screen waiting for input) is ruled out:
- The routine's exit doesn't go to user input or input-poll code — it goes to soft-reset.
- The kick sites set values like `0xA0`, `0xD0`, `0x200`, `0x10` — these are "time remaining" deadlines, not input-state flags.
- The decrement-and-return pattern at `0x0003A186` doesn't gate on input, it's a pure countdown.

### §5.2 OPEN-001 and OPEN-004 collapse

This finding makes OPEN-001 and OPEN-004 the same root cause:

- **OPEN-004 (bootstrap re-entry):** the watchdog timing out and soft-resetting IS the bootstrap re-entry mechanism. Every ~4.3s cycle, the system: bootstraps → does some init → fails to reach watchdog kicks → watchdog expires → ~3.6s delay → soft-reset back to bootstrap.
- **OPEN-001 (rendering broken):** the arcade game-loop never advances to the post-init state that writes nametable cells (Plane A/B at `0xC000`/`0xE000`). This is the SAME reason it never advances to the watchdog kicks. Both manifestations have the same underlying cause: arcade game-loop main path is blocked somewhere upstream of both nametable composition and watchdog kicking.

**Recommended formal disposition:** OPEN-001 and OPEN-004 share a single root cause. Tracking them as separate issues is fine for evidence-trail purposes (their independent symptoms motivated independent investigations), but their resolution is single. Routing call (separate-with-cross-reference vs. formal merge) is Chad's.

### §5.3 Why the kicks aren't reached — most plausible hypothesis

Andy's hypothesis (supported by but not proven by static analysis): the arcade game-loop is gated on the bootstrap re-entry path itself. Specifically:

- Bootstrap (`_bootstrap` at `0x00000202`, per `boot.s:124` and verify_rastan_direct_boot_guard.py:11) runs Genesis-side init (VDP, palette, tile preload, scene_load).
- Bootstrap then jumps to arcade code at `_bootstrap`'s exit (`jmp (0x00003A200).l` per `boot.s:166`).
- The arcade code at `0x00003A200` is supposed to run the arcade main loop, including reaching the kick sites.
- Something in the arcade main-loop path early on (between `0x00003A200` and `0x0003A5D4`+) hits the watchdog routine at `0x0003A180` before reaching any kick. Since the watchdog timer at `0x00FF002C` is whatever bootstrap initialized it to (likely zero from BSS clear at `_bootstrap_clear_staging` — `boot.s:175` clears `palette_dirty` and other state; if `0x00FF002C` is in the BSS-cleared range, it starts at zero).
- First call to watchdog routine: `%a5@(44)` is zero → delay-and-reset.

This would mean every bootstrap cycle hits the watchdog ROUTINE before any code KICKS it. The dance never gets past the first watchdog call.

To confirm this hypothesis, the next investigation needs to determine: between `_bootstrap` exit (jump to `0x00003A200`) and the first call to `watchdog_or_reset` (at `0x0003A180`), is there any code path that should reach a kick site? If yes, why isn't it? If no, the watchdog routine was designed to be called from a context where SOME prior code path had set `0x00FF002C` to non-zero — and that prior context is missing.

### §5.4 Recommended next diagnostic step

Andy recommends (Chad routes):

**Cross-reference with arcade Rastan source.** The arcade-original MAME `rastan.cpp` or local `docs/reference/mame/rastan/` (if it contains arcade disassembly) should show what arcade source `0x00039F80..0x00039FAC` (= the routine's arcade origin, per identity_offset 0x200) is supposed to do, what calls it, and what kick site is supposed to set `0x00FF002C` (= arcade workram `0x002C` offset from `%a5`) before the first call. This is static-evidence work that can be done without runtime traces.

**Alternative:** bookmark cycle (BM-NNN, post-OPEN-013 closure) at a strategic point to capture whether the bootstrap's exit to arcade code (`0x00003A200`) ever advances past, say, `0x00003A4xx` before falling into the watchdog. This requires the bookmarks_v2 mechanism to be implementation-complete (CLOSED-009 re-verification per OPEN-013).

**Alternative:** static control-flow trace from `_bootstrap_clear_staging` exit (post-bootstrap) through the JMP at `boot.s:166` to `0x00003A200` and forward, identifying the first call to `0x0003A180`. This is a multi-hour static analysis but is feasible from the post-patch disassembly.

Each alternative produces different evidence with different cost. Chad routes.

### §5.5 Architectural significance

This finding is a meaningful narrowing of OPEN-001/OPEN-004's investigation. Before this work, the proximate cause was "arcade game-loop blocked somewhere"; that "somewhere" is now narrowed to "before the first kick-site is reached after bootstrap." Andy's nametable-composition-path classification identified `0x055948` as a downstream consequence of the same blockage; this investigation identifies `0x0003A5D4` (and the other kicks) as upstream consequences. Together, the picture is "arcade execution after bootstrap fails to advance past the kick-site range, repeats every ~4.3s."

This is closer than the project has been to the gating instruction. The next diagnostic step's purpose is to identify exactly where the post-bootstrap arcade path diverges from what arcade Rastan expects.

## §6 Phase 6 — Record corrections

These are notes in this design doc, NOT modifications to BM-002, BM-003, or CLOSED-010/CLOSED-011 records.

### §6.1 BM-002 characterization update

The BM-002 investigation report (`docs/design/Andy_BM002_runtime_failure_investigation.md`) characterized the code at `0x0003A19C` as "delay loop / soft-reset preamble." **That reading was correct** — the bytes Andy verified in Phase 1 of this investigation match the BM-002 reading, and the structural role (delay loop ending in soft-reset) is exactly what BM-002 described.

What this investigation ADDS is the broader context: the routine at `0x0003A180` (which the delay loop is part of) is the OPEN-004 bootstrap-re-entry MECHANISM. BM-002's correctness given the evidence available at the time stands; this investigation's incremental contribution is recognizing the same code as the system's primary re-entry driver, not just a static-disassembly curiosity.

The BM-002 report does NOT need to be modified. This design doc supplements it.

### §6.2 BM-002 trace "7 hits in 30 seconds" reinterpretation

BM-002's MAME trace (`mame_run_log.txt`) reported 7 hits at `pc=03a19c` over a 30-second observation. The Build 0077 register-window video shows 84.65% of frames at the loop addresses over 14.8 seconds — meaning the CPU is constantly in this region, not visiting it briefly seven times.

**The 7-hit number is OPEN-014-style undersampling of a saturated loop.** Per OPEN-014's parked-helper sampling-gap finding, the MAME tracer's sampling condition (or its breakpoint-on-instruction mechanism) undersamples tight loops. The same effect applies here: the tracer hit the BNE.S 7 times when the CPU was actually executing it billions of times.

This reinterpretation doesn't change BM-002's classification — the BM-002 fault (Class A address-space mismatch) is still the correct classification. What changes is the interpretation of trace evidence about how often `0x0003A19C` was reached. It was reached constantly; the trace just samples it sparsely.

### §6.3 BM-003 Outcome A "almost immediate helper park" clarification

BM-003 produced Outcome A — the activator at runtime PC `0x0003A19C` fired and the helper at `0x00071C78` parked, validating the bookmarks_v2 model. The MAME trace showed the helper hit "almost immediately."

**Reinterpretation:** the activator fired immediately because the CPU is CONSTANTLY at `0x0003A19C` in the watchdog delay loop. Any cycle in which the bookmark activator is present at `0x0003A19C` will divert PC to the helper on the very next iteration of the loop — which happens within microseconds of any execution moment.

The bookmarks_v2 model is still validated by BM-003 (the bookmark fired correctly under the validated coordinate model). What clarifies is the INTERPRETATION of the speed: the helper park was instant not because `0x0003A19C` is reached briefly during normal post-boot flow, but because the CPU is essentially always there. Any future bookmark at a "non-saturated" target (e.g., a kick site) will likely produce a slower or absent helper park — and that's the actual reachability signal for OPEN-001/OPEN-004 evidence.

The CLOSED-010 / CLOSED-011 closure records do NOT need to be modified. The closure conditions (positive-control Outcome A + new schema test matrix pass) are still met. This is interpretation enrichment, not invalidation.
